✅ Step 1 — AI Vision & Requirements

Vision ที่มีอยู่จริง (ยืนยันจาก BLUEPRINT.md/CLAUDE.md): "AI by Default" — ใช้ AI ลดงานซ้ำของ Admin/Lister ทุกที่ที่ปลอดภัยจะทำได้: (1) AI Quick Add — สร้างประกาศจากรูป/เสียง/PDF (2) AI Concierge/ContactRail Chatbot — ตอบลูกค้าแทน Admin

Requirements ยืนยันจาก Source: Chatbot ต้องตอบจากข้อมูลจริงเท่านั้น (ห้ามมโนราคา/สถานะ), รองรับ 8 ภาษา auto-detect, ปฏิเสธ Restricted Topics อย่างสุภาพ; AI Quick Add ต้องสร้าง SEO title/description/keywords/photo captions ที่ใช้ได้จริงเป็น alt-text

✅ Step 2 — AI Architecture Design

Architecture ที่ใช้จริง: Client (ContactRail.dc.html / AI Quick Add.dc.html) → window.claude.complete() (เฉพาะ preview environment นี้) หรือ Cloud Function claudeComplete (production) → Anthropic API (server-side secret ผ่าน Firebase Secret Manager) → ตอบกลับ client

Data Context Injection: Chatbot ดึง CURRENT LISTINGS + AI Notes + AI Persona + AI Restrictions จาก Firestore ผ่าน firebase-client.js แล้วฉีดเข้า system prompt ทุกครั้งที่ถาม — ไม่มี fine-tuning หรือ vector DB ใช้ prompt injection แบบ context-window ล้วน (เหมาะกับขนาดข้อมูลปัจจุบัน 24 properties)

ข้อจำกัดที่ทราบ: window.claude.complete ทำงานเฉพาะใน design-preview environment นี้เท่านั้น — Production ต้องผ่าน Cloud Function เสมอ (มี fallback logic อยู่แล้วในโค้ด ContactRail.dc.html บรรทัด 480-483)

✅ Step 3 — AI Provider and Architecture — Evidence Audit

Provider ที่ใช้งานจริง: Anthropic Claude เท่านั้น — ยืนยันจาก functions/index.js (fetch("https://api.anthropic.com/v1/messages"), header x-api-key จาก ANTHROPIC_API_KEY secret)

ไม่พบ OpenAI หรือ Gemini — ค้นหาทั้ง Repository (keyword: openai, gemini, OPENAI_API_KEY, GEMINI_API_KEY) ไม่พบการเชื่อมต่อหรือ dependency ใดๆ ทั้งสิ้น — ยืนยันว่า Claude เป็น Provider เดียวจริง ไม่มี integration ที่ active หรือ dormant หลงเหลืออยู่

Production Secrets เป็น Server-side จริง: ANTHROPIC_API_KEY ใช้ defineSecret() จาก firebase-functions/params เก็บใน Firebase Secret Manager — เรียกด้วย .value() เฉพาะภายใน Cloud Function เท่านั้น

ไม่พบ API Secret รั่วไหลใน Client Files — ตรวจสอบ ContactRail.dc.html และ AI Quick Add.dc.html แล้วไม่พบ API key ใดๆ ฝัง hardcode ทั้ง 2 ไฟล์เรียกผ่าน Cloud Function URL เท่านั้น

แยก Preview-only Code จาก Production Code ชัดเจน: ContactRail.dc.html บรรทัด 480 มีเงื่อนไข if (window.claude && window.claude.complete) — ใช้ได้เฉพาะใน design-preview environment นี้เท่านั้น; หากไม่มี window.claude (กรณี Production จริง) จะ fallback ไปเรียก Cloud Function URL โดยตรงเสมอ (บรรทัด 483)

Model ที่ใช้แยกตามงาน: claude-haiku-4-5 (chat แบบ real-time, max_tokens 600) และ claude-sonnet-4-5 (AI Quick Add ร่างประกาศ, max_tokens 4096)

✅ ContactRail AI Chatbot — Evidence Audit (เสริม Step 6)

Source File: ContactRail.dc.html (ยืนยันโดยตรง)

ใช้ข้อมูลจริง: ดึง CURRENT LISTINGS ผ่าน getEffectiveProperties() ฉีดเข้า system prompt ทุกครั้งที่ถาม — system prompt สั่งชัดเจนว่าตอบข้อเท็จจริง (ราคา/ห้อง/สถานะ/รหัสทรัพย์) จาก CURRENT LISTINGS context เท่านั้น ห้ามมโน

ภาษาที่รองรับ: 8 ภาษา (EN/TH/RU/ZH/DE/NO/FR/IT) ยืนยันจาก CHAT_I18N object จริง — auto-detect ภาษาจากข้อความล่าสุดของลูกค้าเสมอ

AI Persona/Restrictions: ดึงจาก Firestore ผ่าน fetchAiPersona()/fetchAiRestrictions()/fetchAiNotes() จริง — persona name/role default เป็น "Anna" หากยังไม่ตั้งค่า

Restricted-topic Handling: system prompt สั่งให้ปฏิเสธหัวข้อต้องห้ามอย่างสุภาพ redirect ไปติดต่อแทน โดยไม่เปิดเผยว่ามีข้อจำกัดอยู่ (ยืนยันจาก prompt text จริง)

ไม่มโนราคา/สถานะ/ทำเล/ข้อมูลทรัพย์: ยืนยันจาก system prompt บรรทัด FACTS ที่บังคับใช้เฉพาะ CURRENT LISTINGS เท่านั้น

Escalation/Contact Path: ยืนยัน 2 กลไกจริง — (1) เมื่อ reply มีคำว่า "contact us"/"ติดต่อเรา" ฯลฯ (regex ตรวจ 7 ภาษา) ระบบแสดงปุ่ม "Contact us" ที่เปิด Inquiry Form ทันที (2) เมื่อเกิด Error จาก AI Service ระบบแสดงข้อความ "Sorry, something went wrong. Please try again or contact us directly." พร้อมปุ่ม Contact us เสมอ — ไม่ปล่อยให้ลูกค้าติดค้างโดยไม่มีทางออก

✅ Step 4 — AI Service Layer (API Integration)

Cloud Function เดียวรองรับ 2 รูปแบบ Call (functions/index.js claudeComplete): (1) Chat mode — รับ {system, messages} ส่งตรงไป Messages API (2) Structured mode — รับ content + tool schema บังคับ Claude ตอบเป็น JSON ที่ validate โดย Anthropic เอง (ใช้ใน AI Quick Add เพื่อไม่ให้เกิด malformed JSON)

Error Handling: try/catch ครบทั้ง 2 mode, ส่ง error message ที่มีประโยชน์กลับ client (ไม่ leak API key หรือ stack trace ดิบ)

Timeout/Memory: timeoutSeconds: 300, memory: 512MiB — เพียงพอสำหรับ Sonnet ตอบยาวสุด 4096 tokens

✅ Step 6 — AI Listing Assistant (AI Quick Add) — Evidence Audit

Source File: AI Quick Add.dc.html (ยืนยันโดยตรง)

Workflow จริง: อัปโหลดรูป (multi-select) → บันทึกเสียง (hold-to-talk, transcript แก้ไขได้ เพิ่มได้เรื่อยๆ) → แนบ PDF/รูป project doc (เสริม) → ใส่พิกัด Google Maps (decimal lat,lng) → กด "ให้ AI ร่างประกาศ" → รีวิว/แก้ไข draft → กรอกข้อมูลเจ้าของ → เผยแพร่จริง

Generated Fields (structured tool schema, บังคับ format โดย Anthropic เอง): title, shortDesc, fullDesc, seoKeywords, photoCaptions (array, 1 ต่อรูป), internalNotes (แยกข้อมูลลับ เช่น ชื่อโครงการ/developer ไม่ให้ขึ้น public)

SEO Title/Description/Keywords/Photo Captions: ยืนยันจาก prompt จริง — title เป็น SEO headline ผสมไทย/อังกฤษตาม pattern ที่คนค้นจริง, seoKeywords 10-14 คำ, photoCaptions ออกแบบเป็น '{{"{keyword-rich phrase}"}} {{"{{CODE}}"}}' สลับภาษาไทย/อังกฤษ 50/50

Data Validation/Error Handling: fbCaptionError/refineError state มี UI แสดง error พร้อมปุ่ม "ลองใหม่"; รูปภาพต้องมีอย่างน้อย 1 รูปก่อน generate; ไม่พบ validation ฝั่ง client สำหรับพิกัด (lat,lng format) — ต้องพึ่งผู้ใช้กรอกถูกรูปแบบเอง เป็น Known Gap เล็กน้อย

Alt Text: ยังเป็น Backlog — AI สร้าง photoCaptions ที่ออกแบบให้ใช้เป็น alt-text ได้อยู่แล้ว (มี {"{{CODE}}"} placeholder) แต่ ยังไม่ได้เชื่อมกับ <img alt> จริงบนหน้าเว็บไซต์ public — ยืนยันตรงกับ CLAUDE.md §24.8 ที่บันทึกไว้ก่อนหน้านี้

Nearby POI: ใช้ Google Places + Distance Matrix API จริง (client-side, key จำกัดด้วย HTTP referrer) ค้นหา 7 ประเภทสถานที่ แล้วคำนวณระยะทาง/เวลาขับรถจริง — ไม่ได้ mock

Facebook Post: AI ร่าง caption แยกจาก footer คงที่ (contact block + hashtag ที่ Admin ตั้งใน Site Content), Copy-to-clipboard พร้อมใช้

✅ Step 7 — AI Content Generation (SEO / Translation / Description)

SEO Content Generation: ยืนยันจาก prompt จริงใน AI Quick Add.dc.html — title ออกแบบเป็น SEO headline ตาม pattern ที่คนค้นจริง (ผสมไทย/อังกฤษ), photoCaptions ออกแบบให้ใช้เป็น alt-text ได้ทันที (มี placeholder {"{{CODE}}"} ให้ template แทนรหัสทรัพย์จริง), สุ่มภาษาไทย/อังกฤษ 50/50 ต่อ caption เพื่อดึง traffic ทั้ง 2 กลุ่ม

Translation: ไม่พบ AI-powered translation pipeline สำหรับเนื้อหาเว็บไซต์ในโค้ดปัจจุบัน — 8 ภาษาของเว็บไซต์ (data.js i18n dictionary) เป็นการแปลที่เตรียมไว้ล่วงหน้า (static translations) ไม่ใช่ AI แปลสด ส่วนที่ AI แปลจริงมีเพียง Chatbot ที่ auto-detect ภาษาผู้ใช้และตอบภาษาเดียวกัน (translation แบบ conversational เท่านั้น ไม่ใช่แปลเนื้อหาเว็บไซต์)

Description Generation: ยืนยันแล้วใน Step 6 — AI Quick Add สร้าง shortDesc/fullDesc พร้อม keyword natural

Known Gap ที่มีอยู่แล้ว (บันทึกไว้ก่อนหน้านี้ใน CLAUDE.md §24.8): AI สร้าง photoCaptions/alt-text ไว้แล้ว แต่ ยังไม่ได้เชื่อมกับ <img alt> จริงบนหน้าเว็บไซต์ public — เป็นช่องว่าง SEO ที่มีมูลค่าสูงสุดที่ยังไม่ปิด

✅ Step 8 — AI Testing & Implementation Review

Prompt Safety ตรวจสอบจากโค้ดจริง: Chatbot system prompt บังคับ (1) ตอบข้อเท็จจริงจาก CURRENT LISTINGS เท่านั้น ห้ามมโน (2) ปฏิเสธ Restricted Topics แบบสุภาพ ไม่เปิดเผยว่ามีข้อห้าม (3) ตอบภาษาเดียวกับที่ผู้ใช้พิมพ์เสมอ (4) จำกัดความยาวคำตอบ 2-5 ประโยค

AI Quick Add Privacy Rule: prompt สั่งชัดเจนห้ามเปิดเผยชื่อโครงการ/developer ใน public fields (title/shortDesc/fullDesc/photoCaptions) — ให้เก็บไว้ใน internalNotes เท่านั้น

ข้อจำกัด: นี่คือ Static Prompt/Code Review ยืนยันว่า Safety Rule ถูกเขียนไว้ในโค้ดถูกต้อง — ไม่ใช่ Live Adversarial Testing (ทดลองยั่วให้ AI ทำผิดกฎจริงบน Production) ซึ่งต้องทำโดยเจ้าของหรือ QA ที่มีสิทธิ์เข้าถึงระบบจริง

✅ Step 9 — AI Cost, Safety & Reliability — Evidence Audit

Usage Controls ที่มีอยู่จริง: max_tokens กำหนดไว้ชัดเจนทั้ง 2 mode (600 chat / 4096 Quick Add) ป้องกัน runaway cost ต่อ 1 request; Model Selection คุม cost ที่ระดับ architecture (Haiku ถูก/เร็วสำหรับ chat volume สูง, Sonnet เฉพาะ Quick Add ที่ใช้ไม่บ่อย)

Rate-limit/Spending-control สถานะปัจจุบัน: Backlog — ยังไม่ implement ยืนยันจาก functions/index.js ไม่พบ rate-limiter หรือ App Check middleware ใดๆ ในทั้ง claudeComplete function

🔖 Chatbot Spending Cap: Backlog (ตามคำสั่ง Product Owner ให้ทำเครื่องหมายชัดเจน) — เจ้าของเคยถามเรื่องนี้มาก่อนแล้ว ยังไม่มี hard cap หรือ rate-limit ระดับผู้ใช้/วัน

Failure Handling เมื่อ AI Service ไม่พร้อมใช้งาน: ยืนยันจาก ContactRail.dc.html catch block จริง — แสดงข้อความ "Sorry, something went wrong.
