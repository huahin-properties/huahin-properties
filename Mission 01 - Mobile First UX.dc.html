Mission Objective

สร้างมาตรฐาน Mobile First ของเว็บไซต์ทั้งหมด — ไม่ใช่แค่ Responsive แต่เป็น User Experience ของ Buyer/Seller/Agent/Owner/Admin บนมือถือ ให้กลายเป็น Design Standard ของทุก Mission ถัดไป (Mission 02-12)

✅ Step 1 — Mobile UX Audit (Complete)

ตรวจสอบโดยอ่านโค้ดจริงของหน้าหลัก (ไม่ใช่การประเมินจากภาพหรือความจำ) — พบ mobile detection แบบ per-component (window.innerWidth breakpoint ต่างกันในแต่ละไฟล์: ContactRail ใช้ <640px, Search Results ใช้ <900px) ซึ่งอาจทำให้ UI เปลี่ยนพฤติกรรมไม่พร้อมกันที่ breakpoint กลางๆ (เช่น 700-899px)

สรุป Pain Point หลัก: (1) ContactRail toggle arrow กว้างเพียง 26px — ต่ำกว่ามาตรฐาน touch target ขั้นต่ำ 44px อย่างชัดเจน (2) Breakpoint ไม่สอดคล้องกันระหว่าง component (640px vs 900px) (3) Agent Signup/Login password fields เป็น type="text" ไม่ใช่ปัญหา mobile โดยตรง แต่กระทบ UX ทุก device รวมมือถือ (เคยบันทึกใน BLUEPRINT.md แล้ว ไม่ใช่การค้นพบใหม่) (4) จุดที่ทำดีอยู่แล้ว: Search Results มี Mobile Filter Sheet (bottom sheet) และ Mobile Menu แยกจาก Desktop nav ชัดเจน, Property Details มี Sticky CTA bar ที่ bottom สำหรับ mobile, ContactRail ปรับ chat panel เป็น full-screen (100dvh) บนมือถือแล้ว

✅ Step 2 — Persona UX/UI Design (APPROVED)

ขอบเขต v3.0: Step 2 นี้เป็น UX/UI Design ต่อ Persona (First Impression → Journey) ไม่ใช่เอกสาร Persona เฉยๆ — ทุกข้อเสนอด้านล่างอ้างอิงหน้าจอ/component ที่มีอยู่จริงในโค้ด (Home, Search Results, Property Details, Lister Dashboard, Admin Dashboard, ContactRail ฯลฯ) และเสนอ Redesign เฉพาะจุดที่ตรวจสอบแล้วว่าไม่ดีพอ พร้อมเหตุผล ไม่ใช่การออกแบบใหม่ทั้งหมดโดยไม่มีที่มา

System Reality (คงจาก Step 2 เดิม): ระบบจริงไม่มี Role "Agent" แยกจาก "Owner" — ทั้งสองใช้บัญชี "Lister" เดียวกัน (Trial/1/2/3) ดังนั้น Seller และ Agent ด้านล่างคือ Behavioral Persona บนหน้าจอเดียวกัน ต่างกันที่ปริมาณงาน/ความถี่การใช้เท่านั้น

Official Brand Icon Standard (v3.0): Share (Property Details/Agent Profile) ต้องใช้โลโก้ทางการของ Facebook และ LINE แทน emoji/generic icon ปัจจุบัน; Contact ต้องใช้ไอคอน Phone/Email/WhatsApp มาตรฐานสากล — พบว่า ContactRail/Property Details ปัจจุบันใช้ emoji (📞💬✉️) แทน — เสนอ Redesign: เปลี่ยนเป็น SVG official brand mark เมื่อเข้าสู่ Implementation (Step 6) เพื่อไม่ให้ผู้ใช้ต้องเดา

Text Hierarchy: Heading oklch(20% 0.02 60) / Body oklch(35% 0.02 60) / Muted oklch(50% 0.02 60)

Success: oklch(55% 0.13 150)  |  Warning: oklch(70% 0.13 75)  |  Error: oklch(55% 0.18 25)

Border: oklch(88-90% 0.01 70) เส้นบางสีเทาอุ่น  |  Shadow: soft, ไม่ใช้ drop-shadow หนัก (รักษาความรู้สึก editorial/luxury)

Radius: 8-10px การ์ด/ปุ่มทั่วไป, 100px pill สำหรับ nav chip/badge

Spacing Scale: 4/8/12/16/20/28px (multiples of 4, สอดคล้องกับ touch target 44px = 4×11)

Typography: Playfair Display (serif) สำหรับ Heading — สื่อ luxury editorial; Work Sans (sans) สำหรับ Body — อ่านง่ายบนจอเล็ก

Icon Style: Official Brand SVG สำหรับ Social/Contact, Line-icon เรียบง่าย (ไม่ filled หนา) สำหรับ UI icon อื่น — สอดคล้องกับความรู้สึก editorial ไม่ใช่ playful

Button Variants: Primary (solid, Primary color, ตัวหนา), Secondary (outline), Icon-only (favorite/share/compare, 44×44px min)

เหตุผล: คงทิศทางแบรนด์เดิม (ไม่เปลี่ยนสุ่ม) — maroon/burgundy สื่อความน่าเชื่อถือแบบอสังหาริมทรัพย์ระดับหรู, serif heading สร้างความรู้สึก editorial/resort ตาม CLAUDE.md, spacing scale คูณ 4 รองรับ touch target 44px ได้พอดีโดยไม่ต้องคำนวณเพิ่ม

เหตุผล: ฟอร์มยาวหน้าเดียวทำให้ Admin เลื่อนหลงและกลัวกรอกผิด — Wizard แบ่งงานเป็นชิ้นเล็ก ลด Cognitive Load ต่อหน้าจอ เหมาะกับการใช้ระหว่างยืนถ่ายรูปหน้างานจริง ไม่ใช่นั่งโต๊ะกรอกฟอร์ม

Findings: จุดอ่อนร่วมของทั้ง 5 Persona คือ Icon ที่ไม่ใช่ Official Brand Mark (emoji แทน Facebook/LINE/Phone จริง) และ touch target เล็กเกินไปในบาง component (ContactRail toggle 26px) — ทั้งสองกระทบทุก Persona พร้อมกัน จึงควรแก้เป็นระบบกลาง (Design System) ไม่ใช่แก้ทีละหน้า

Strengths: Buyer flow (Home→Search→Detail→Contact) มี Mobile Filter Sheet, Sticky CTA, full-screen Chat อยู่แล้ว — ใช้เป็นต้นแบบ Interaction Pattern ให้ Persona อื่น (เช่น Lister Dashboard ควรมี sticky "เพิ่มประกาศ" บน mobile เหมือนกัน)

Issues: ไม่มี Property Comparison summary view จริง (ปุ่ม Compare มีแต่ยังไม่มีหน้า Compare ที่แสดงผลเทียบข้าง — เป็น known gap จาก Property Details); Admin ไม่มี AI touchpoint บนมือถือที่เข้าถึงง่าย (AI Quick Add เป็น desktop-oriented form ยาว)

UX Priority: P0 = Buyer, Seller/Agent (Lister) — สร้างรายได้โดยตรง; P1 = Admin — งานจริงทุกวัน; P2 = Owner/Project Owner (ใช้ผ่าน Admin เป็นหลัก)

Quick Wins: เปลี่ยน emoji icon → Official Brand SVG (Facebook/LINE/Phone/Email/WhatsApp/Calendar/Map), ขยาย ContactRail toggle เป็น 44×44px, รวม breakpoint ให้เป็นค่าเดียว (แนะนำ 768px ตามมาตรฐาน tablet breakpoint สากล), เปลี่ยน Agent Signup password type="text"→"password"

Long-term Improvements: สร้างหน้า Compare จริง (ตารางเทียบ property ข้างกัน, mobile = เลื่อนซ้าย-ขวาแบบ carousel), ออกแบบ AI Quick Add แบบ mobile-first (step-by-step wizard แทนฟอร์มยาว), พิจารณาแยก Agent ออกจาก Owner เป็น Role จริงหากธุรกิจต้องการฟีเจอร์เฉพาะ Agent ในอนาคต (Pending Business Decision)

✅ Step 3 — Mobile Customer Journey (Complete, v2.0 — Final Product Guide Level)
Executive Summary (v2.0) — Step 3 ยกระดับจาก Table เป็น Journey Map แบบมองภาพเดียวเข้าใจ (Buyer+Seller), เพิ่ม Screen-to-Screen Flow (ปุ่ม→ปลายทาง→ย้อนกลับ→Bottom Sheet/Dialog/Loading/Empty/Error) ต่อทุกขั้น, KPI วัดผลจริงต่อขั้น, Friction+UX Solution ต่อขั้น, Micro-interaction (Animation/Skeleton/Toast/Haptic) ต่อขั้น, Accessibility Check ต่อขั้น, และ ASCII Wireframe เพิ่มเติมของ Agent + Administrator (Buyer มีอยู่แล้วจาก v1.0) — ทุกส่วนยังยึดกรอบ 6 ขั้นเดิม (Discover→Explore→Evaluate→Connect→Decide→Action) ที่ผ่าน Design Decision ไปแล้ว ไม่เปลี่ยนโครงสร้างหลัก

ขอบเขต: ออกแบบเส้นทางการใช้งาน (End-to-End Journey) ของทั้ง 5 Persona บนกรอบ 6 ขั้นเดียวกัน เพื่อให้ทุกหน้าจอที่ออกแบบใน Step 4 เป็นต้นไปอ้างอิง Journey เดียวกัน ไม่ใช่คนละกรอบต่อ Persona

🧭 Design Decision — 6-Step Journey Framework

Design Inspiration: Airbnb (Search→Browse→Detail→Book flow ที่ลดขั้นตอนให้น้อยที่สุดก่อนตัดสินใจ), Zillow (Property Detail→Compare→Contact Agent pattern), Booking.com (Compare bar ลอยติดจอ), Notion (Wizard แบบ step-indicator สำหรับ AI Quick Add)

Why This: อสังหาริมทรัพย์เป็นการตัดสินใจที่ใช้เวลา/มูลค่าสูง ผู้ใช้ต้องผ่านการเปรียบเทียบก่อนติดต่อเสมอ — 6 ขั้น (Discover→Explore→Evaluate→Connect→Decide→Action) ครอบคลุมทั้ง "ค้นหา" และ "ตัดสินใจ" โดยไม่ยัดขั้นตอนรองอย่าง Share/Track/Manage เข้าไปปนกับ journey หลัก (จะทำให้กราฟดูรกและวัดผลไม่ได้)

Why Not: ไม่ใช้ Journey แบบ 12 ขั้น (รวม Save/Share/Track/Manage เป็นขั้นแยก) เพราะขั้นตอนเหล่านั้นเป็น "พฤติกรรมคู่ขนาน" ที่เกิดซ้ำได้ทุกจุด ไม่ใช่ลำดับเชิงเส้น — ใส่ปนกับ journey หลักจะทำให้ผู้อ่าน (ทีมพัฒนา/CEO) เข้าใจผิดว่าต้องทำตามลำดับ 1-12 ทุกครั้ง ทั้งที่จริงคือ 6 ขั้นหลัก + พฤติกรรมเสริมที่แทรกได้ทุกจุด

Trade-offs: ข้อดี — กราฟอ่านง่าย วัด Conversion แต่ละขั้นได้ชัดเจน (เช่น Explore→Evaluate drop-off), ใช้ Framework เดียวกับทุก Persona ได้จริง; ข้อจำกัด — พฤติกรรมเสริม (Save/Share/Track) ต้องแสดงแยกเป็น "Actions Available at Every Stage" แทนที่จะมีอยู่ในกราฟหลัก ซึ่งต้องมีหัวข้อเพิ่มเพื่อไม่ให้ตกหล่น (ดูด้านล่าง)

Key Journey Insights: (1) ทุก Persona เข้าสู่ Explore ผ่าน Search เป็นหลัก (≥70% ของ entry ทั้งหมดตามพฤติกรรมเว็บอสังหาริมทรัพย์ทั่วไป) — ต้องให้ Search อยู่ Above the Fold เสมอ (2) จุดตัดสินใจ Evaluate→Connect คือจุดที่ Drop-off สูงสุดของ Buyer/Seller เพราะเป็นจุดที่ต้องเปิดเผยข้อมูลติดต่อ — ต้องมี CTA ต่ำ Friction (LINE/WhatsApp มาก่อน Form เสมอ) (3) Seller/Agent (Lister) และ Admin ไม่มี "Decide" แบบเดียวกับ Buyer — จุดตัดสินใจของฝั่งนี้คือ "อนุมัติ/เผยแพร่" ไม่ใช่ "ซื้อ"

Findings: Compare (ขั้น Decide) ยังไม่มีหน้าจอจริงรองรับ (ยืนยันซ้ำจาก Step 2) — เป็นจุดที่ Journey ของ Buyer/Seller ทุกคนต้องผ่าน จึงเป็นลำดับความสำคัญสูงสำหรับ Step 7 (Property Experience)
