Design Philosophy Report v1

เหตุผลเบื้องหลังการออกแบบระบบ — ทำไมถึงเลือกทางนี้ ไม่ใช่ทางอื่น
อัปเดตล่าสุด: 17 กรกฎาคม 2569 · อ่านคู่กับ System Architecture Report v1

ข้อจำกัดตั้งต้นที่กำหนดทุกการตัดสินใจ

ก่อนจะอธิบายทีละส่วน ต้องเข้าใจ 3 ข้อจำกัดที่ยึดเป็นแกนกลางของทุกทางเลือกทางเทคนิคในระบบนี้:

  1. เจ้าของเว็บเขียนโค้ดเองไม่ได้และไม่มีเครื่องมือพัฒนาในเครื่องส่วนตัว — ทุกอย่างต้อง deploy ได้ผ่านการอัปโหลดไฟล์ธรรมดา ไม่ใช่ผ่าน command line ที่ต้องติดตั้งอะไรเพิ่ม
  2. ธุรกิจยังเล็ก รายได้ยังไม่แน่นอน — ทุกบริการที่เลือกต้องมี free tier ที่ใช้ได้จริงในสเกลนี้ ไม่ใช่บริการที่บังคับจ่ายรายเดือนก้อนใหญ่ตั้งแต่วันแรก
  3. ต้องแข่งกับพอร์ทัลอสังหาฯ ที่มีอยู่แล้วในตลาดด้วย "ความเร็วในการเห็นผล" — ฟีเจอร์ต้องขึ้นจอทันทีที่พิมพ์เสร็จ ไม่ใช่รอ build/compile ก่อนถึงจะเห็นอะไรเลย เพราะการวนซ้ำเร็วคือสิ่งที่ทำให้ทันคู่แข่งที่มีทีมพัฒนาเต็มรูปแบบได้

ทุกหัวข้อด้านล่างคือคำตอบของคำถามเดียวกัน: "ทางเลือกนี้เข้ากับ 3 ข้อจำกัดข้างบนได้ดีที่สุดหรือไม่"

ทำไมเลือก Firebase

Firebase ถูกเลือกไม่ใช่เพราะ "ดีที่สุดในทางเทคนิค" แต่เพราะเป็นตัวเดียวที่ตอบโจทย์ทั้ง 3 ข้อจำกัดพร้อมกัน:

  • ไม่มีเซิร์ฟเวอร์ให้ดูแล — เจ้าของเว็บไม่ต้องรู้จักคำว่า "เซิร์ฟเวอร์ล่ม" เลย Google ดูแลความพร้อมใช้งานให้ทั้งหมด ตรงกับข้อจำกัดข้อ 1 (ไม่มีคนดูแลระบบประจำ)
  • Free tier ใจกว้างพอสำหรับธุรกิจขนาดนี้ — Firestore/Hosting/Functions ฟรีในปริมาณการใช้งานระดับหลักร้อย-พันผู้ใช้ต่อวัน ตรงกับข้อจำกัดข้อ 2 (ต้นทุนเริ่มต้นต่ำ ขยับตามรายได้จริง)
  • อ่าน/เขียนข้อมูลแบบเรียลไทม์ข้ามหน้าได้ทันที — ทรัพย์ที่แอดมินเพิ่มผ่าน AI Quick Add ปรากฏในหน้า Search Results ทันทีโดยไม่ต้องมีขั้นตอน sync แยก เพราะทุกหน้าอ่านฐานข้อมูลเดียวกันตรงๆ
  • Authentication + Storage + Functions อยู่ในระบบนิเวศเดียวกัน — ไม่ต้องเชื่อมต่อบริการหลายเจ้าที่มีบัญชี/บิลลิ่งแยกกัน ลดจุดที่อาจพังจากความไม่เข้ากันของบริการต่างค่าย

ทางเลือกที่ตัดออก: ฐานข้อมูล SQL ดั้งเดิม (ต้องมีเซิร์ฟเวอร์แยกดูแล ขัดข้อจำกัดข้อ 1), self-hosted บน VPS (ต้องมีคนคอย patch/update ความปลอดภัยเอง)

ทำไมเลือกสถาปัตยกรรม Design Components (หน้าเว็บแยกไฟล์อิสระ)

แทนที่จะสร้างเว็บแบบ "แอปเดียวก้อนใหญ่" ที่ต้อง build ทั้งระบบใหม่ทุกครั้งที่แก้อะไรแม้แต่จุดเดียว ระบบนี้แยกทุกหน้าเป็นไฟล์อิสระของตัวเอง (`.dc.html`) เหตุผล:

  • วาดผลได้ทันทีที่พิมพ์ ไม่ต้องรอ build — ตรงกับข้อจำกัดข้อ 3 (ความเร็วในการวนซ้ำ) เห็นผลลัพธ์แทบจะพร้อมกับที่แก้ไข ไม่ใช่รอ compile หลายสิบวินาที-นาทีทุกครั้ง
  • แก้ไขหน้าหนึ่งไม่เสี่ยงทำหน้าอื่นพัง — เพราะแต่ละหน้าเป็นไฟล์เดี่ยวจบในตัว การแก้ Lister Dashboard ไม่มีทางไปกระทบโค้ดของ Admin Dashboard โดยไม่ตั้งใจ ซึ่งสำคัญมากเมื่อไม่มีชุดทดสอบอัตโนมัติ (automated tests) คอยจับความผิดพลาดแทนคน
  • อัปโหลดไฟล์เดี่ยวทับได้ตรงๆ — ตรงกับข้อจำกัดข้อ 1 เจ้าของเว็บอัปโหลดแค่ไฟล์ที่เปลี่ยนผ่านหน้าเว็บ GitHub ได้เลย ไม่ต้องรันคำสั่ง build ก่อนอัปโหลด
  • อ่านโค้ดแล้วเข้าใจง่ายกว่าเมื่อกลับมาแก้ทีหลัง — ไม่มีเลเยอร์ bundler/webpack/build config ซ่อนอยู่ระหว่างสิ่งที่เห็นบนจอกับโค้ดจริง เวลามีปัญหาก็ไล่ดูตรงจุดได้เลย

แลกมาด้วยอะไร: โค้ดที่ใช้ซ้ำ (เช่น PropertyCard) ต้องแยกเป็นไฟล์ย่อยแล้ว "import" เข้ามาใช้ แทนที่จะแชร์กันแบบ library ทั่วไปในแอปก้อนใหญ่ — ยอมรับข้อเสียนี้ได้เพราะแลกกับความเรียบง่ายและความเร็วในการแก้ไขที่ได้กลับมามีค่ากว่ามากในบริบทนี้

ทำไมใช้ Stripe

การเงินคือจุดที่พลาดไม่ได้ที่สุดในทั้งระบบ หลักคิดคือ "ให้ผู้เชี่ยวชาญด้านการเงินทำงานหนักแทนเรา แทนที่จะสร้างระบบรับเงินเอง":

  • ไม่ต้องเก็บเลขบัตรเครดิตเอง — ตัดความเสี่ยงด้านกฎหมาย/มาตรฐานความปลอดภัย (PCI compliance) ที่ระบบเล็กๆ ไม่มีทางทำได้เองอย่างถูกต้องครบถ้วน
  • Subscription + webhook + customer portal มาสำเร็จรูป — เรื่องยากอย่าง "จะทำยังไงถ้าบัตรหมดอายุกลางเดือน" หรือ "ลูกค้าอยากยกเลิกเอง" Stripe จัดการให้หมดโดยไม่ต้องเขียนโค้ดเพิ่ม
  • Ledger (`payments`) เขียนได้เฉพาะฝั่งเซิร์ฟเวอร์ผ่าน webhook ที่ตรวจลายเซ็นแล้วเท่านั้น — กันไม่ให้ใครปลอมแปลงว่า "จ่ายเงินแล้ว" ได้จากฝั่งเบราว์เซอร์
  • Dynamic pricing (Featured/VIP/แบนเนอร์) — ใช้ price_data สร้างราคาแบบไดนามิกได้โดยไม่ต้องเข้าไปสร้าง Product ใน Stripe Dashboard ทุกครั้งที่ราคาปรับ ต่างจาก subscription (แพ็กเกจ 1/2/3) ที่ราคาคงที่จึงสร้าง Product ล่วงหน้าไว้ — เลือกวิธีที่เหมาะกับแต่ละประเภทการขาย ไม่ใช้วิธีเดียวตายตัวทุกกรณี

ทำไมแบ่งระบบสมาชิก/แพ็กเกจแบบนี้

โครงสร้าง "ทดลองฟรี 30 วันครั้งเดียว + 3 แพ็กเกจจ่ายเงิน + เตรียมที่ไว้ให้ VIP ในอนาคต" ไม่ใช่ตัวเลขที่สุ่มเลือก แต่ออกแบบตามหลักจิตวิทยาการตัดสินใจซื้อ:

  • ทำไมไม่มีแพ็กเกจฟรีถาวร — แพ็กเกจฟรีถาวรสร้างกลุ่มผู้ใช้ที่ไม่มีวันจ่ายเงินและทำให้เว็บดู "มีของค้าง" จากคนที่ไม่ตั้งใจทำธุรกิจจริงจัง การจำกัดเป็น "ทดลองครั้งเดียว 30 วัน" บังคับให้ทุกคนต้องตัดสินใจในที่สุดว่าจะจ่ายต่อหรือไม่ ทำให้ฐานผู้ใช้ที่เหลืออยู่มีคุณภาพสูงกว่า
  • ทำไม trial ได้สิทธิ์เทียบเท่าแพ็กเกจสูงสุด — หลักการ "ให้ชิมของดีที่สุดก่อน" คนที่เห็นผลลัพธ์เต็มรูปแบบ (ป้ายแนะนำ+หน้าแรก) จะรู้สึกถึงคุณค่าที่เสียไปตอนหมด trial ชัดเจนกว่าคนที่ไม่เคยเห็นเลย — สร้างแรงจูงใจอัปเกรดจาก "loss aversion" (กลัวเสียของที่เคยมี) ซึ่งได้ผลกว่าการโฆษณาบอกคุณสมบัติเฉยๆ
  • ทำไมฟีเจอร์พื้นฐานเท่ากันทุกแพ็กเกจ ต่างแค่โควตา+ตัวเร่งมองเห็น — ถ้าตัดฟีเจอร์พื้นฐานออกตามแพ็กเกจ ลูกค้าจะรู้สึกว่า "ของที่เคยใช้ได้ถูกยึดคืน" เวลาไม่อัปเกรด แต่การจำกัดแค่ปริมาณ (จำนวนประกาศ) และการมองเห็น (Featured) ทำให้พวกเขารู้สึกว่ายังใช้งานได้เต็มที่ แค่ "โตช้ากว่า" — ความรู้สึกนี้ต่างกันมากในเชิงจิตวิทยาการตลาด
  • ทำไมล็อกฟีเจอร์ที่ยังไม่มีสิทธิ์ไว้แบบสีเทาแทนการซ่อน — คนที่ไม่เห็นฟีเจอร์เลยจะไม่มีวันรู้ว่ามีสิ่งนั้นอยู่ ไม่มีวันอยากอัปเกรดเพื่อสิ่งที่ไม่รู้จัก การโชว์แบบล็อก (สีเทา+ไอคอนกุญแจ) คือการโฆษณาแบบไม่ต้องพูดอะไรเลย — เห็นแล้วอยากได้เอง
  • ทำไมต้องมีคิวอนุมัติประกาศของแอดมิน — ทดลองฟรีที่ไม่มีการตรวจสอบเลยจะกลายเป็นช่องให้สแปม/ประกาศหลอกลวงหลุดขึ้นเว็บทันที ซึ่งทำลายความน่าเชื่อถือของเว็บทั้งเว็บจากประกาศเดียว การอนุมัติก่อนขึ้นจริงคือ "ด่านคุณภาพ" ที่ต้นทุนถูกที่สุด (เวลาแอดมินไม่กี่นาทีต่อประกาศ) เทียบกับความเสียหายที่ป้องกันได้
  • ทำไมประกาศมีอายุ 30 วันแล้วซ่อนอัตโนมัติ — เว็บอสังหาฯ ที่มีประกาศเก่าค้างเป็นปีดูเหมือน "เว็บร้าง" ทันทีที่ลูกค้าสังเกตเห็นวันที่โพสต์เก่ามาก การบังคับหมดอายุทำให้เว็บดู "มีชีวิต" อยู่เสมอ โดยไม่ลบข้อมูลทิ้งจริง (แค่ซ่อน) เพื่อให้เจ้าของกลับมาต่ออายุได้ทันทีโดยไม่เสียงานที่กรอกไว้

ทำไม AI ทำงานแบบนี้ (ไม่ใช่แบบอื่น)

  • ทำไม AI ต้องตอบจากฐานข้อมูลจริงเท่านั้น ห้ามมโน — อสังหาริมทรัพย์คือการตัดสินใจเรื่องเงินก้อนใหญ่ ถ้า AI ตอบราคาหรือสถานะผิดแม้แต่ครั้งเดียว ความเสียหายต่อความน่าเชื่อถือจะมากกว่าประโยชน์ที่ AI ตอบได้ลื่นไหลกว่า จึงยอมให้ AI "ตอบได้แคบกว่ามนุษย์" เพื่อแลกกับ "ไม่มีวันโกหกลูกค้า"
  • ทำไมมีหัวข้อต้องห้ามที่ AI ปฏิเสธแบบไม่บอกว่ามีข้อจำกัด — ถ้า AI พูดตรงๆ ว่า "ฉันถูกห้ามไม่ให้ตอบเรื่องนี้" ลูกค้าจะสงสัยว่าเว็บปิดบังอะไรอยู่ การเบี่ยงไปหาช่องทางติดต่อแทนอย่างสุภาพรักษาประสบการณ์ที่ดีไว้ได้โดยไม่เปิดเผยข้อมูลที่ไม่ควรเปิดเผย (เช่น ค่าคอมมิชชัน)
  • ทำไมแปลภาษาแค่ครั้งเดียวตอนบันทึก ไม่แปลสดทุกครั้งที่มีคนเปิดดู — การเรียก AI มีต้นทุนต่อครั้ง ถ้าแปลสดทุกครั้งที่มีคนเข้าดูหน้าเดียวกัน ต้นทุนจะเพิ่มตามจำนวนผู้เข้าชมแบบไม่มีเพดาน แต่เนื้อหาทรัพย์ไม่ได้เปลี่ยนบ่อย จึงแปลครั้งเดียวเก็บผลลัพธ์ไว้ใช้ซ้ำได้ตลอดไปจนกว่าจะแก้ไขเนื้อหาใหม่ — ประหยัดต้นทุนได้มากโดยคุณภาพไม่ต่างกัน
  • ทำไมจำกัดจำนวนทรัพย์ที่ส่งให้ AI อ่านต่อข้อความแชท — เมื่อทรัพย์ในระบบเพิ่มเป็นหลักร้อย การส่งทั้งหมดให้ AI อ่านทุกข้อความจะทำให้ต้นทุนต่อข้อความสูงขึ้นเรื่อยๆ ไม่มีเพดาน การกรองเฉพาะที่เกี่ยวข้องกับคำถามจึงเป็นการควบคุมต้นทุนล่วงหน้าก่อนที่จะเป็นปัญหาจริง ไม่ใช่แก้ทีหลังตอนบิลมาแพงแล้ว
  • ทำไม AI Quick Add ให้แอดมินตรวจ/แก้ก่อนบันทึกเสมอ ไม่บันทึกอัตโนมัติ — AI ร่างเนื้อหาได้เร็วแต่ไม่รับประกันความถูกต้อง 100% การให้มนุษย์เป็นด่านสุดท้ายก่อนเผยแพร่รักษาคุณภาพของทุกประกาศไว้ได้ในขณะที่ยังประหยัดเวลาร่างเนื้อหาไปมากกว่า 90%

ทำไมฐานข้อมูลออกแบบแบบนี้

  • ทำไมรูปภาพแยกเป็นคอลเลกชันของตัวเอง (`propertyPhotos`) แทนที่จะฝังในเอกสารทรัพย์ — ฐานข้อมูล Firestore จำกัดขนาดเอกสารแต่ละใบไว้ที่ 1MB ทรัพย์ที่มีรูปเยอะจะชนขีดจำกัดนี้ทันทีถ้าฝังรวมกัน การแยกคนละเอกสารทำให้แต่ละรูปไม่มีเพดานจำนวนที่แท้จริงจากข้อจำกัดนี้
  • ทำไมย้ายไฟล์รูปจริงไปที่ Storage แทนที่จะเก็บเป็น base64 ในฐานข้อมูล — การฝังรูปเป็นข้อความยาวๆ ในฐานข้อมูลทำให้ทุกครั้งที่ดึงรายการทรัพย์มาแสดง ต้องโหลดข้อมูลรูปทั้งหมดมาด้วยแม้ยังไม่เห็นภาพ ทำให้หน้าเว็บช้า การเก็บที่ Storage แล้วอ้างอิงแค่ลิงก์สั้นๆ ทำให้โหลดเร็วขึ้นมากโดยเฉพาะหน้าที่แสดงทรัพย์หลายรายการพร้อมกัน
  • ทำไมไม่ลบข้อมูลทรัพย์ทิ้งเวลาหมดอายุ/ปิดการขาย — ข้อมูลประวัติ (ราคาที่เคยตั้ง, รายละเอียดเดิม) มีค่าในการอ้างอิงย้อนหลังและวิเคราะห์แนวโน้มตลาดในอนาคต การซ่อนจากเว็บสาธารณะเพียงพอต่อจุดประสงค์ "ไม่ให้เว็บดูร้าง" แล้ว ไม่จำเป็นต้องทำลายข้อมูลถาวรซึ่งกู้คืนไม่ได้
  • ทำไมสิทธิ์การเขียนข้อมูล (Security Rules) เข้มงวดถึงระดับฟิลด์ ไม่ใช่แค่ระดับเอกสาร — สมาชิกแต่ละคนต้องแก้ไขได้เฉพาะทรัพย์ของตัวเอง และห้ามเปลี่ยนสถานะเป็น "กำลังแสดง" เองโดยไม่ผ่านอนุมัติ กฎที่ตรวจสอบถึงระดับฟิลด์ (ไม่ใช่แค่ "ใครเป็นเจ้าของเอกสาร") คือสิ่งเดียวที่ป้องกันไม่ให้คนหลบเลี่ยงขั้นตอนอนุมัติผ่านการเขียนข้อมูลตรงได้

ทำไม UX เป็นแบบนี้

  • ทำไม 8 ภาษาตั้งแต่วันแรก ไม่ใช่แค่ไทย+อังกฤษ — หัวหิน/ชะอำ/ปราณบุรีเป็นตลาดที่มีผู้ซื้อชาวรัสเซีย/จีน/ยุโรปจำนวนมากซื้ออสังหาฯ จริงจัง การรองรับภาษาเหล่านี้ตั้งแต่ต้นคือความแตกต่างที่จับต้องได้จากพอร์ทัลอื่นที่มักมีแค่ไทย/อังกฤษ ไม่ใช่ฟีเจอร์เสริมที่เพิ่มทีหลังได้ง่ายๆ เพราะกระทบทุกหน้า/ทุกฟอร์ม
  • ทำไมแชทบอทมีเมนูปุ่มด่วนตั้งแต่เปิดมาครั้งแรก — ผู้ใช้จำนวนมากไม่รู้จะเริ่มพิมพ์อะไร การให้ทางลัด (ฝากขาย/หาเช่า/สมัคร Agent) ลดอุปสรรคแรกที่ทำให้คนเลิกใช้งานก่อนได้ลองจริง
  • ทำไมป็อปอัพดูทรัพย์ในหน้าเว็บสมาชิกไม่พาออกไปหน้าอื่น — ทุกครั้งที่ผู้ใช้ต้องเปลี่ยนหน้า คือจุดที่อาจเสียผู้ใช้ไปเลย (bounce) การให้ดูรายละเอียดในหน้าเดียวกันรักษาผู้ใช้ไว้ในบริบทเดิม เพิ่มโอกาสที่จะดูทรัพย์หลายรายการต่อการเข้าใช้งานครั้งเดียว
  • ทำไมมีโหมดปิดปรับปรุงเว็บไซต์ (Maintenance Gate) — เจ้าของเว็บต้องการดูงานระหว่างพัฒนาได้จริงบนโดเมนจริง โดยที่ลูกค้าทั่วไปยังไม่เห็นเวอร์ชันที่ยังไม่สมบูรณ์ — แก้ปัญหา "ทดสอบบนของจริงแต่ไม่กระทบภาพลักษณ์" โดยไม่ต้องมีระบบ staging แยกที่ซับซ้อนกว่า

ทำไม SEO เป็นแบบนี้

  • ทำไมให้ความสำคัญกับ SEO สูงสุดตั้งแต่ต้น — ธุรกิจนี้ไม่มีงบโฆษณาก้อนใหญ่เท่าพอร์ทัลรายใหญ่ การค้นหาแบบออร์แกนิก (คนพิมพ์หา "บ้านหัวหิน" ใน Google) คือช่องทางลูกค้าที่ต้นทุนต่อรายต่ำที่สุดในระยะยาว จึงต้องลงทุนเวลาทำให้ถูกต้องตั้งแต่โครงสร้างแรก แก้ทีหลังยากกว่ามาก (ต้องรื้อ URL ที่เคยติดอันดับแล้ว)
  • ทำไมทุกโพสต์ Facebook/TikTok ที่ AI ร่างมีลิงก์กลับเว็บเสมอ — ทุกลิงก์ที่ถูกแชร์ต่อคือ "backlink" ที่ช่วยอันดับค้นหาสะสมไปเรื่อยๆ โดยไม่มีต้นทุนเพิ่ม ยิ่งสมาชิกโพสต์เยอะ เว็บยิ่งได้ประโยชน์ทาง SEO แบบทวีคูณโดยอัตโนมัติ ไม่ต้องทำการตลาดเพิ่ม
  • ทำไมต้องมี URL ถาวรต่อทรัพย์ + Structured Data (JSON-LD) — Google อ่านหน้าเว็บที่มีข้อมูลระบุชัดเจน (ราคา/ที่อยู่/ประเภท) ได้แม่นยำกว่าข้อความธรรมดา ทำให้มีโอกาสขึ้นแสดงผลแบบ Rich Result (เช่น โชว์ราคาในผลค้นหาเลย) ซึ่งเพิ่มอัตราการคลิกได้มากกว่าลิงก์ธรรมดา
