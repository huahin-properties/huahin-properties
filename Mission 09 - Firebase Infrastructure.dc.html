✅ Step 2 — Firestore Database Structure Audit

Collections พบใน firestore.rules (17 collections, ยืนยันจาก match statement จริงทุกอัน): properties, propertyPhotos, siteContent (+ projectDashboard/projectDashboard_backup_v1 sub-cases), profilePhotos, aiNotes, banners, externalServices, buyerRegistrations, adminUsers, staffInvites, leads, payments, listers, owners, tenants, settings, listerLeads

Naming: camelCase สม่ำเสมอทุก collection ไม่พบชื่อไม่ตรงรูปแบบ

Relationships: propertyPhotos↔properties ผ่าน field ที่เชื่อมด้วย listerId ownership check; buyerRegistrations↔listers ผ่าน listerId; listerLeads↔listers ผ่าน listerId — ทุกความสัมพันธ์ enforce ที่ระดับ Rules ไม่ใช่แค่ระดับ UI

Indexes: ไม่พบไฟล์ firestore.indexes.json ในโปรเจกต์ — หมายความว่ายังไม่มี composite index ที่ประกาศไว้ชัดเจน (Firestore auto-index เฉพาะ single-field query; query ที่ซับซ้อนกว่านั้นบน production จะ throw error พร้อมลิงก์สร้าง index อัตโนมัติจาก Firebase Console — ยังไม่เคยเห็น error แบบนี้ในโค้ด แปลว่า query ปัจจุบันทั้งหมดยังอยู่ในระดับที่ไม่ต้องการ composite index)

Scalability: Default-deny + per-collection allow เป็นรูปแบบที่ scale ได้ปลอดภัย ไม่มี wildcard เปิดกว้างที่พบ

✅ Step 4 — Firestore Security Rules Audit

Current Rules: Deny-by-default (บรรทัดสุดท้ายของไฟล์คือ match /{document=**} {"{"} allow read, write: if false {"}"}) — ทุก collection ที่ไม่ได้ประกาศไว้ explicit จะถูกปฏิเสธทั้งหมด

Role Model: isAdmin()/isOwner() ตรวจ request.auth.uid ตรงกับ hardcoded Owner UID หรือ exists() lookup ใน adminUsers — ไม่มี client ฝั่งไหนตั้งค่า role ของตัวเองได้

Least Privilege: ตรวจสอบทีละ collection พบรูปแบบที่เหมาะสม เช่น payments (allow write: if false — เขียนได้เฉพาะฝั่ง server/Function เท่านั้น), settings/siteContent sensitive docs (isAdmin/isOwner only), leads (create: if true, read: isAdmin only — ป้องกันข้อมูลติดต่อรั่วไหล)

Documentation Consistency: comment ในไฟล์ rules เองอธิบายเหตุผลของแต่ละกฎละเอียด (เช่นเหตุผลที่ storage rules หลวมกว่า Firestore rules สำหรับ propertyPhotos) — สอดคล้องกับ BLUEPRINT.md §5/§11

สรุป: ไม่พบข้อบกพร่องที่มีหลักฐานเพียงพอให้แก้ Rules ในรอบนี้ — Rules คงเดิม

✅ Step 6 — Cloud Storage Rules

storage.rules ตรวจสอบจริง: propertyPhotos/ และ profilePhotos/ — read: public (จำเป็นสำหรับแสดงรูปบนเว็บไซต์), write: isAdmin() หรือ signed-in user ใดก็ได้ (comment ในไฟล์อธิบายเหตุผลชัดเจน: Storage rules parse propertyId จาก filename ไม่ได้แม่นยำเท่า Firestore rules จึงจงใจหลวมกว่าเล็กน้อย โดยพึ่ง Firestore doc's listerId check เป็นการควบคุมที่แท้จริง)

Fallback path ({"{"}allPaths=**{"}"}): read: public, write: isAdmin() only — ปลอดภัยเป็น default สำหรับ path อื่นที่ไม่ได้ระบุ

ข้อสังเกต (ไม่ใช่ข้อบกพร่อง): write: request.auth != null (ไม่เช็ค ownership เจาะจง) หมายความว่า lister ที่ signed-in คนหนึ่งสามารถ upload ไปที่ path ของ property คนอื่นได้ทาง Storage โดยตรง — แต่ Firestore doc ที่เชื่อมรูปเข้ากับ property ยังคง enforce listerId ownership อยู่ จึงไม่เกิดความเสียหายจริงในทางปฏิบัติ (ไฟล์ orphan ใน Storage ที่ไม่มี Firestore doc อ้างอิงจะไม่แสดงผลที่ไหนเลย) — บันทึกเป็น Known Risk ระดับต่ำ ไม่ใช่ Launch Blocker

✅ Step 7 — Indexes & Query Performance

Indexes: ไม่มี firestore.indexes.json — ทุก query ที่มีอยู่ในโค้ดปัจจุบันเป็น single-field หรือใช้ client-side filter/sort หลังโหลดข้อมูล (ตรวจสอบจาก getEffectiveProperties() pattern ใน data.js/firebase-client.js) จึงไม่ต้องพึ่ง composite index

Slow Query Risk: หากอนาคตเพิ่ม server-side query ที่ filter+sort พร้อมกันหลาย field (เช่น area + price range + beds ในคำสั่งเดียวผ่าน Firestore query แทน client filter) จะต้องสร้าง composite index ก่อน มิฉะนั้น production จะ throw error — เป็นความเสี่ยงที่ควรระวังเมื่อ Mission 02 (Search & Filters) ขยายเป็น server-side query ในอนาคต

Optimization ปัจจุบัน: Client-side filter บน dataset ขนาดเล็ก (24 sample properties) ทำงานได้เร็วพอ — ยังไม่มีปัญหาประสิทธิภาพที่วัดได้จริงในสภาพแวดล้อมนี้ (ต้องวัดจริงบน production กับข้อมูลจริงจำนวนมากกว่านี้ — ดู Step 11)

🔵 Step 8 — Hosting / Domain / SSL / Environment — READY FOR VERIFICATION

พบจาก source: firebase.json ประกาศเฉพาะ firestore/storage/functions — ไม่มี "hosting" key เลย ยืนยันว่าโปรเจกต์นี้ deploy static site ผ่าน GitHub Pages ไม่ใช่ Firebase Hosting (สอดคล้องกับ CLAUDE.md ที่ระบุ "Static site changes go live automatically (GitHub Pages/Hosting)")

Domain/SSL: การตั้งค่า custom domain (huahin.properties) และ SSL certificate เป็นการตั้งค่าฝั่ง GitHub Pages/DNS registrar ไม่ใช่ Firebase Console — ต้องให้เจ้าของยืนยันสถานะจริงของ domain/SSL ที่ใช้งานอยู่ (ไม่มีไฟล์ config ในโปรเจกต์นี้ที่ยืนยันได้)

Secrets: functions/index.js ใช้ defineSecret() จาก firebase-functions/params สำหรับ ANTHROPIC_API_KEY, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET — เก็บใน Firebase Secret Manager ไม่ hardcode ในโค้ด (ตรงตามมาตรฐานความปลอดภัย) แต่ต้องเจ้าของยืนยันผ่าน Firebase Console ว่า secret values ถูกตั้งค่าจริงและไม่หมดอายุ

Environment: apiKey/authDomain/projectId ใน firebase-client.js เป็นค่า public client config ตามปกติของ Firebase Web SDK (ไม่ใช่ความลับ ปลอดภัยที่จะ commit ได้) — ไม่พบปัญหา
